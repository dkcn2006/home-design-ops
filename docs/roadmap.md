# Roadmap

这份路线图用于把 `home-design-ops` 从“项目档案 MVP”推进为一个面向家装设计团队的、类似 Linear 工作方式的项目管理系统。

产品不应该变成通用任务管理工具。它的核心优势应该来自家装业务上下文：项目空间、设计阶段、版本资产、客户确认、报价联动、施工交底和责任人闭环。

## 产品方向

构建一个安静、清晰、可执行的运营工作区，让家装团队可以快速回答五个问题：

- 当前有哪些活跃项目？
- 哪些空间或房间被阻塞？
- 每个空间处在哪个阶段？
- 下一步动作由谁负责？
- 哪些客户确认、图纸、报价或现场问题正在影响推进？

交互模型可以参考 Linear，但领域模型必须保持家装行业特征。

## 当前基础

已经实现：

- 基于 Next.js 的前端应用，包含首页工作区、角色工作台、销售线索页、项目档案页和客户门户。
- 基于 NestJS 的 REST API，目前使用内存 demo 数据。
- 共享 TypeScript 领域模型。
- 项目档案数据已经覆盖客户、线索、需求单、空间、设计版本、效果图、施工图、报价、变更、里程碑、巡检、确认记录和附件。
- 已有面向销售、设计师、项目经理的基础角色任务流。
- 已有 AI 编排桩实现，用于需求提取、布局建议、效果图建议、图纸校核和巡检摘要。

主要缺口：

- 当前工作仍然围绕角色工作台和项目档案组织，还没有形成完整的项目管理系统：缺少空间阶段、个人负责人、任务依赖和清晰生命周期。

## 北极星模型

长期对象模型应该围绕这些实体：

- `Project`：一个家装项目。
- `ProjectSpace`：项目内的空间或环境，例如客厅、厨房、卧室、卫生间、阳台、全屋收纳。
- `WorkflowPhase`：可复用的阶段模板，例如需求确认、平面方案、SU 模型、效果图、施工图、报价、交底、施工跟进、验收。
- `ProjectTask`：类似 Linear issue 的执行单元，属于一个项目，并可选绑定空间和阶段。
- `User`：真实人员，不能只停留在角色层面。
- `TaskAssignment`：任务负责人或负责人变更历史。
- `TaskComment`：讨论、交接和处理记录。
- `TaskDependency`：任务之间的阻塞和依赖关系。
- `TaskLinkedEntity`：任务与设计版本、效果图、施工图、报价、变更、确认记录、巡检或附件之间的关联。

## Phase 1：家装任务核心

目标：把当前 `WorkItem` 升级为真正的任务模型，支持项目、空间、阶段、负责人、状态和优先级。

范围：

- 新增用户、空间、阶段和项目任务的共享领域类型。
- 保留当前角色字段，同时增加 `assigneeId`，让任务可以分配到具体人员。
- 给任务增加 `spaceId` 和 `phaseId`。
- 增加符合家装流程的任务状态：
  - `backlog`
  - `todo`
  - `in_progress`
  - `blocked`
  - `waiting_client`
  - `waiting_internal`
  - `done`
  - `canceled`
- 新增销售、设计师、深化设计、项目经理和管理员的 demo 用户。
- 扩展 seed 数据，加入多个空间和阶段化任务。
- 暴露任务列表、项目任务看板、个人任务收件箱相关 API。

建议接口：

- `GET /projects/:id/tasks`
- `GET /projects/:id/task-board`
- `GET /tasks/my?assigneeId=...`
- `PATCH /tasks/:id/status`
- `PATCH /tasks/:id/assignee`

验收标准：

- 一个项目可以按空间和阶段展示任务。
- 一个任务可以分配给具体人员。
- 一个用户可以看到自己跨项目的任务。
- 现有项目档案页和角色工作台仍然可用。

## Phase 2：项目看板体验

目标：创建第一版类似 Linear 的项目管理视图。

范围：

- 新增 `/projects/[id]/board`。
- 看板按 `ProjectSpace` 分组，并使用阶段列或阶段区块组织任务。
- 任务卡片保持紧凑、清晰、可执行，展示：
  - 标题
  - 状态
  - 优先级
  - 负责人
  - 截止日期
  - 关联资产或确认记录
- 增加项目级进度摘要：
  - 总任务数
  - 阻塞任务数
  - 等待客户确认数
  - 逾期任务数
  - 存在阻塞的空间
- 从项目档案页增加进入项目看板的入口。
- 视觉风格遵循 `DESIGN.md`：温暖、文档化、低噪音、接近 Notion 的运营工具感。

验收标准：

- 看板能清楚说明哪个空间被阻塞以及阻塞原因。
- 页面在桌面端和移动端都可用。
- 看板不替代项目档案，而是作为执行视图补充项目档案。

## Phase 3：我的工作与团队工作台

目标：让责任归属从角色层面推进到具体人员层面。

范围：

- 新增 `/my-work` 或 `/tasks`。
- 按截止日期、状态和项目展示个人任务。
- 将角色工作台升级为使用真实分配任务，而不是只依赖角色型 `WorkItem`。
- 增加团队过滤：
  - 负责人
  - 角色
  - 项目
  - 状态
  - 优先级
  - 空间
  - 阶段
- 增加逾期视图和阻塞视图。

验收标准：

- 设计师可以只看到分配给自己的设计任务。
- 项目经理可以看到所有活跃项目中的阻塞任务。
- 销售不需要逐个项目查找，就能看到客户确认相关任务。

## Phase 4：家装流程模板

目标：让项目初始化和空间任务创建更可复用。

范围：

- 按空间类型定义可复用流程模板。
- 厨房示例阶段：
  - 需求确认
  - 平面方案
  - SU 模型
  - 效果图
  - 柜体深化
  - 水电点位确认
  - 报价冻结
  - 施工交底
- 卫生间示例阶段：
  - 需求确认
  - 平面排布
  - 防水与排水确认
  - 材料确认
  - 施工图
  - 现场验收
- 创建项目或新增空间时，可以生成默认阶段和起始任务。
- 任务允许关联需求单、图纸、报价和确认记录。

验收标准：

- 新增厨房空间时可以生成实用的默认流程。
- 生成的任务可以编辑、分配，并出现在项目看板中。
- 模板能减少手工配置，但不会隐藏项目自身差异。

## Phase 5：客户确认闭环

目标：把任务进度和客户侧确认真正联动起来。

范围：

- 任务可以标记为需要客户确认。
- 任务可以关联 `ConfirmationRecord`。
- 增加 `waiting_client` 状态流转。
- 在客户门户中展示客户待确认事项。
- 在内部视图中展示确认阻塞信息：
  - 哪个任务被阻塞
  - 哪个资产正在等待确认
  - 谁需要跟进
  - 已等待多久

验收标准：

- 创建确认记录后，相关任务可以进入 `waiting_client` 状态。
- 客户在门户中确认或驳回后，相关任务上下文会同步变化。
- 销售和项目经理可以在工作台中看到确认阻塞。

## Phase 6：版本、报价与变更控制

目标：减少由于设计版本、报价版本和变更记录不清晰导致的返工。

范围：

- 任务可以关联设计版本、效果图版本、施工图版本、报价和变更单。
- 任务卡片展示当前依据的版本。
- 当任务依赖已归档或未确认版本时给出提醒。
- 报价差异或设计变更发生时，可以创建或关联后续任务。
- 关键阶段流转需要保留决策历史。

验收标准：

- 项目经理可以判断施工任务基于哪个施工图版本。
- 设计师可以看到哪个客户确认阻塞了设计变更。
- 报价更新可以创建或关联后续跟进任务。

## Phase 7：持久化与权限

目标：从 demo 数据推进到真实多用户应用。

范围：

- 扩展 Prisma schema，加入用户、任务、阶段、分配、评论、依赖和关联实体。
- 将内存仓储写操作替换为 PostgreSQL 服务。
- 增加认证，并支持基于当前登录用户的视图。
- 增加基础权限：
  - 管理员可以管理全部工作。
  - 销售可以管理线索和客户跟进。
  - 设计师可以管理设计任务和设计资产。
  - 项目经理可以管理交付任务、巡检和交底。
  - 客户只能看到客户侧确认数据。
- 增加 migration 和 seed 脚本。

验收标准：

- demo 数据可以写入 PostgreSQL。
- 创建和更新任务后，服务重启仍然保留数据。
- 个人任务收件箱来自真实登录身份。

## Phase 8：AI 辅助运营

目标：只在能减少重复协同成本的地方使用 AI。

范围：

- 从需求单生成任务草稿。
- 根据空间类型建议阶段检查清单。
- 汇总阻塞任务，生成每日运营摘要。
- 检查施工图交底任务是否缺少必要确认。
- 生成客户友好的确认摘要。
- 从会议纪要或巡检记录中提取行动项。

验收标准：

- AI 输出必须结构化、可审阅。
- AI 不允许静默修改项目状态。
- AI 建议任务必须经过用户确认后才能加入项目。

## 建议实施顺序

近期建议按这个顺序推进：

1. 增加共享任务、用户、阶段和看板类型。
2. 扩展 seed 数据，加入真实感更强的用户、空间、阶段和任务。
3. 增加项目任务看板和个人任务收件箱的仓储方法与 API。
4. 实现 `/projects/[id]/board`。
5. 从项目档案页和首页增加看板入口。
6. 将角色工作台升级为消费真实分配任务。
7. 增加客户确认与任务的关联。
8. demo 流程稳定后再扩展 Prisma schema。

## 产品原则

- 项目档案仍然是单一事实来源。
- 看板用于执行，不用于承载所有项目事实。
- 工作要分配到具体人员，同时保留角色上下文。
- 客户确认是一等阻塞因素。
- 设计版本和报价不是普通附件，而是可关联的运营记录。
- 状态数量要少，但含义要清楚。
- UI 保持安静、结构化、适合日常运营。
